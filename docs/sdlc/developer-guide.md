## Purpose

This guide is for MedScribe-R-Us engineers. It explains what the security
pipeline does, how to read a finding, how to handle false positives correctly,
and when to escalate to the security team.

The goal is not to make security someone else's problem. It's to make it easy
to do the right thing at the point in the development cycle where it costs least.

---

## What Runs and When

| Tool | When | What It Checks |
|---|---|---|
| **Gitleaks** (pre-commit hook) | Before every commit | Secrets in staged files |
| **Gitleaks** (CI) | Every push | Secrets in full git history |
| **Semgrep** (custom rules) | Every PR | PHI-in-logs, missing auth, LLM output handling |
| **Semgrep** (community) | Every PR | OWASP Top 10, JWT issues, common Python/JS vulns |
| **pip-audit / npm audit** | Every PR | Dependency CVEs |
| **Trivy** | Push to `main`/`staging` | Container image CVEs + misconfigurations |
| **OWASP ZAP** | Post-deploy to staging | Live application security scan (3 auth contexts) |

### Setting up the pre-commit hook

```bash
# From the repo root
pip install pre-commit
pre-commit install

# Test it works
pre-commit run --all-files
```

The pre-commit hook runs Gitleaks on staged files before each commit. It takes
~2 seconds and will block a commit if a secret is detected. This is faster and
safer than catching secrets in CI.

## Reading a Semgrep Finding

When Semgrep blocks your PR, the output looks like this:

```
backend/services/transcription.py
  Line 47: phi-variable-in-logger
  Severity: ERROR
  Message: Potential PHI variable passed to logger — HIPAA violation risk (T-006).
  ...
  ❯ logger.info("Transcription complete", transcript=transcript_data)
```

**Fields to pay attention to:**

- **Rule ID** — links to the rule file in `semgrep-rules/` for full context
- **Severity** — `ERROR` blocks the PR; `WARNING` creates an issue
- **Message** — explains the specific risk and what to do
- **Line** — exact location; check the surrounding context before acting

**What to do:**

1. Read the flagged line in context. Is there actually PHI in that variable?
2. If yes — fix it. Options: remove the log call, log a redacted version, or
   log only a non-PHI identifier (session UUID, not patient data).
3. If no — suppress it with a justification comment (see below).

## Suppressing a False Positive

If Semgrep flags something that is genuinely not a risk, suppress it — but
document why.

**Python:**
```python
# nosemgrep: phi-variable-in-logger
# Justification: `session_id` is a UUID generated by MedScribe.
# It contains no patient data. Confirmed by code review.
logger.info("Session started", session_id=session_id)
```

**What makes a valid justification:**
- Explains specifically why the flagged pattern is safe in this context
- References the variable's type or source ("UUID generated by MedScribe")
- Is written for a future reader, not just to clear the gate

**What is not a valid justification:**
- `# nosemgrep: phi-variable-in-logger` with no comment
- "False positive" with no explanation
- "Will fix later"

Suppressions are reviewed quarterly. Invalid suppressions are removed and
the finding is re-opened.

## Reading a Trivy Finding

Trivy reports show up as PR comments (delta report) and in the GitHub
Advanced Security code scanning dashboard.

```
HIGH: CVE-2024-12345
Package: cryptography 41.0.3
Fixed In: 41.0.4
Description: Timing side-channel in RSA decryption...
```

**Action by severity:**

| Severity | Action | Timeframe |
|---|---|---|
| Critical | Update package immediately | Before merge |
| High | Update package | Before merge to `main` |
| Medium | Create tracking issue | 30 days |
| Low | Backlog | 90 days |

**How to fix a Trivy Python finding:**

```bash
# Update the specific package
pip install --upgrade cryptography==41.0.4

# Regenerate the lock file
pip freeze > requirements.txt

# Verify the CVE is resolved
trivy image medscribe-backend:local --severity HIGH,CRITICAL
```

**Suppressing a Trivy finding** (only when no fix exists):

Add to `.trivyignore` at the repo root:

```
# CVE-2024-12345
# Justification: Affects RSA ProxyManager — not used in MedScribe.
# Re-review: 2025-01-01
CVE-2024-12345
```

Suppressions without justification comments will be removed.

## Reading a ZAP Finding

ZAP findings appear as GitHub Issues automatically created after each
staging deployment. High severity findings also block the next environment
promotion.

```
[DAST] Missing Content Security Policy — Medium
Source: OWASP ZAP authenticated scan (clinician role)
CWE: CWE-693
Description: Content Security Policy (CSP) header not set...
SLA: 30 days
```

**How to investigate:**

1. Check the ZAP report artifact in the workflow run for the full request/response
2. Reproduce locally against the staging environment using the test token
3. Identify whether it's a genuine finding or a scan artifact

**Common ZAP false positives for MedScribe:**
- "Cookie No HttpOnly Flag" on WebSocket session cookies — known accepted risk
  (documented in `.zap/rules.tsv`)
- "X-Content-Type-Options not set" on `/health` endpoint — acceptable; no body returned

## When to Escalate to the Security Team

Escalate immediately (don't wait for a sprint) if:

- Gitleaks detects a secret — assume compromised, rotate before anything else
- Trivy finds a Critical CVE in a package that handles PHI
  (MongoDB driver, FHIR client, JWT library, cryptography)
- ZAP finds a High severity finding involving authentication or authorization
- You're unsure whether a suppression is appropriate
- A security finding touches the PHI scrubbing layer, the approval gate,
  or the FHIR write-back path

**Contact:** Open a GitHub Issue with the `security-review-requested` label,
or reach out directly to the security team lead. Do not post credential or
finding details in Slack public channels.

## Security Review Request Process

For changes that affect:
- Any new API endpoint or change to an existing endpoint's auth mechanism
- A new external integration or data store
- A new user role or permission scope
- The PHI scrubbing layer
- The AI summarization pipeline
- The EMR Integration Service

...open a security review request before shipping to production. Create a
GitHub Issue with the `security-review-requested` label and include:

- What is changing
- Which data flows are affected (reference L1 DFD flow numbers)
- What new attack surface is introduced
- Your proposed mitigations

The security team will respond within 2 business days for non-urgent requests.

## The Security Golden Rules

1. **PHI never goes in logs.** Ever. Not in debug, not in exceptions, not in request echoing.
2. **Every FastAPI route that touches PHI requires an auth dependency.** No exceptions.
3. **LLM output is untrusted** until it clears the Output Validation Service. Do not read fields from it before that.
4. **Secrets go in Secret Manager.** Never in `.env` files, never hardcoded, never in comments.
5. **If you're unsure, ask.** The cost of a 10-minute security question is much lower than the cost of a HIPAA breach.
